### ISSUE: Concurrency Guard Double Decrement in `buildPlugin`
**Severity**: high
**Location**: `buildPlugin`
**Intent**: The `activeOps` counter should accurately reflect the number of ongoing operations. The `finally` block in `buildPlugin` is intended to decrement `activeOps` only if `runBuild` was not launched (i.e., an early error occurred). The `runBuild` function's `finally` block is responsible for decrementing `activeOps` once the asynchronous build completes.
**Reality**: The `buildPlugin` function's `finally` block checks `if (!opStarted)` to decide whether to decrement `activeOps`. However, `opStarted` is set to `true` *before* `runBuild` is called, but `runBuild` itself is an `async` function that is `await`ed only for its launch, not its completion. The `finally` block of `buildPlugin` will execute *before* `runBuild` has completed and before `runBuild`'s own `finally` block has had a chance to decrement `activeOps`. This means that if an error occurs *after* `opStarted` is set to `true` but *before* `runBuild`'s `finally` block executes (e.g., an unhandled rejection within `runBuild` itself, or if `runBuild` completes very quickly), the `activeOps` counter will not be decremented by the `buildPlugin`'s `finally` block. The `runBuild`'s `finally` block will then decrement it, leading to a single decrement.

The problem is that the `buildPlugin`'s `finally` block is designed to catch *synchronous* errors before `runBuild` is launched. But `runBuild` is launched, and then `buildPlugin` returns. The `finally` block of `buildPlugin` will execute *immediately* after `runBuild` is launched, not after it completes. This means `activeOps` is incremented, `runBuild` is launched, `buildPlugin`'s `finally` block executes (and does nothing because `opStarted` is true), and then `buildPlugin` returns. The `activeOps` counter remains incremented until `runBuild` eventually completes and its `finally` block executes. This is the *intended* behavior for the success path.

The issue arises if `runBuild` itself throws an error *synchronously* before its `finally` block is registered, or if the `catch` block of `runBuild`'s promise chain is not robust enough. The current logic for `opStarted` and the `finally` block in `buildPlugin` is trying to prevent a double decrement, but the asynchronous nature of `runBuild` means the `finally` block of `buildPlugin` will execute too early to reliably manage the `activeOps` counter for the `runBuild`'s lifecycle.

A simpler and more robust approach would be to increment `activeOps` at the very beginning of `buildPlugin` and then ensure *only one* decrement happens, either in `buildPlugin`'s `catch` block for synchronous errors, or in `runBuild`'s `finally` block for asynchronous completion. The current logic is overly complex and prone to misinterpretation of execution order.

### ISSUE: `runBuild`'s `catch` block does not update `activeOps`
**Severity**: medium
**Location**: `runBuild`
**Intent**: The `activeOps` counter should be decremented when an operation completes, whether successfully or with an error. The `runBuild` function is responsible for decrementing `activeOps` upon its completion.
**Reality**: The `runBuild` function has a `.finally()` block attached to its call in `buildPlugin`. This `.finally()` block is responsible for decrementing `activeOps`. However, the `runBuild` function itself is `async` and has its own `try...catch` block. If an error occurs within `runBuild`'s `try` block, it will be caught by its `catch` block, the build status will be updated to 'failed', but the `activeOps` counter is not decremented *within* `runBuild`. The decrement relies solely on the `.finally()` attached to the `runBuild` promise in `buildPlugin`. This is technically correct because the `.finally()` block *will* execute regardless of success or failure of the promise. However, the comment in `buildPlugin`'s `finally` block suggests a misunderstanding of how `async/await` and `finally` blocks interact with fire-and-forget promises. The `activeOps` decrement is correctly handled by the `.finally()` attached to the `runBuild` promise in `buildPlugin`, but the design intent for `opStarted` in `buildPlugin` is confusing and could lead to future errors if not understood correctly. The issue is more about clarity and potential for future bugs due to the complex `opStarted` logic in `buildPlugin` rather than an immediate bug in `runBuild` itself.
