### BUG: Incorrect AbortController usage in `handleScan`
**Severity**: medium
**Location**: `handleScan`
```typescript

    const data: RepoScanResult = await res.json()
```

**Problem**: The `if (!signal.aborted)` check is performed *after* `await res.json()`. If the signal is aborted while `res.json()` is still pending, the `res.json()` call will throw an `AbortError`, which is caught by the outer `try...catch` block. However, if the signal is aborted *after* `res.json()` completes but *before* the `if (!signal.aborted)` check, the state updates (`setError` or `setScanResult`/`onSkillsFound`) will still occur on a potentially unmounted component or with stale data, which is what the `signal.aborted` check is trying to prevent. The same logic applies to the `setError` call in the `if (!res.ok)` block.

**Fix**: Move the `if (!signal.aborted)` check to immediately after the `await res.json()` call, before any state updates are attempted.

```typescript

        const data = await res.json()
        if (signal.aborted) return; // Add this check
        setError(data.error || 'Failed to scan repository')
        return
    }

    const data: RepoScanResult = await res.json()
    if (signal.aborted) return; // Add this check
    setScanResult(data)
    onSkillsFound(data.skills, url.trim(), ref)
```
