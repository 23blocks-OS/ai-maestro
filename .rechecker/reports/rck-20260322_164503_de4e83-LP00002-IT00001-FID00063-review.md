This means the cache will remain `null` indefinitely until a successful request completes.
**Fix**: Ensure that `cachedSessions` and `cacheTimestamp` are only updated on successful completion of `fetchAllSessions`. If `fetchAllSessions` fails, the cache should remain in its previous state (or be explicitly cleared if desired, but not updated with a failed result).

export async function listSessions(): Promise<{ sessions: Session[]; fromCache: boolean }> {
  const now = Date.now()

  if (cachedSessions && (now - cacheTimestamp) < CACHE_TTL_MS) {
    return { sessions: cachedSessions, fromCache: true }
  }

  if (pendingRequest) {
    // Await the existing pending request
    try {
      const sessions = await pendingRequest;
      return { sessions, fromCache: false }; // Indicate it was not from this call's cache check
    } catch (error) {
      // If pending request failed, clear it and try again
      pendingRequest = null;
      // Fall through to create a new request

    cachedSessions = sessions;

    // A better approach might be to only clear if the promise matches the one we set.

### BUG: Inconsistent `sessionActivity` update in `sendCommand`
**Severity**: low
**Location**: `sendCommand`
**Problem**: In `sendCommand`, `sessionActivity.set(sessionName, Date.now())` is called unconditionally after sending the command. However, if `requireIdle` is true and the session is *not* idle, the function returns an error without updating the activity. This means that a command attempt on a non-idle session will not update its last activity timestamp, potentially leading to a longer perceived idle time than actual. While `requireIdle` prevents the command from being sent, the attempt itself could be considered an activity.
**Fix**: If an attempt to send a command, even if blocked by `requireIdle`, should be considered an activity, then `sessionActivity.set` should be moved to an earlier point. If only successful command sends count as activity, then the current placement is correct, but the problem description highlights a potential logical inconsistency depending on the desired behavior. Assuming any interaction attempt should update activity, move the update before the `requireIdle` check.

```typescript
export async function sendCommand(
  sessionName: string,
  command: string,
  options: { requireIdle?: boolean; addNewline?: boolean } = {}
): Promise<ServiceResult<{ success: boolean; sessionName: string; commandSent?: string; method?: string; wasIdle?: boolean; idle?: boolean; timeSinceActivity?: number; idleThreshold?: number }>> {
  const requireIdle = options.requireIdle !== false
  const addNewline = options.addNewline !== false

  if (!command || typeof command !== 'string') {
    return { error: 'Command is required', status: 400 }
  }

  const runtime = getRuntime()
  const exists = await runtime.sessionExists(sessionName)
  if (!exists) {
    return { error: 'Tmux session not found', status: 404 }
  }

  // Update activity regardless of idle status, as an attempt to interact is activity
  sessionActivity.set(sessionName, Date.now())

  if (requireIdle && !isSessionIdle(sessionName)) {
    // Return ONLY error without data field to avoid ambiguous response
    return {
      error: 'Session is not idle',
      status: 409
    }
  }

  await runtime.cancelCopyMode(sessionName)
  await runtime.sendKeys(sessionName, command, { literal: true, enter: addNewline })

  return { data: { success: true, sessionName, commandSent: command, method: 'tmux-send-keys', wasIdle: true }, status: 200 }
}
```
