### BUG: Potential race condition with `pollFailures` and `pollRef`
**Severity**: medium
**Location**: `handleBuild`, `clearPoll`
**Problem**: The `pollFailures` ref is reset to `0` in `clearPoll`. However, `clearPoll` is called at the beginning of `handleBuild` and also within the polling interval logic. If `handleBuild` is called rapidly multiple times, it will call `clearPoll` which resets `pollFailures`. This could potentially mask repeated failures if a new build is started before the previous polling interval has had a chance to accumulate 5 failures and trigger the error state. While `pollRef.current` is cleared, `pollFailures.current` is also reset, which might not be the intended behavior if the user is trying to build multiple times quickly and the underlying issue persists.
**Fix**: The `pollFailures` counter should be tied to a specific build attempt. Since `pollRef` is already being cleared, the current implementation is mostly safe from a "leak" perspective. However, to prevent `pollFailures` from being reset prematurely by a new build attempt before the previous one has fully resolved its polling, it's better to ensure `pollFailures` is reset only when a *new* build process truly starts, not just when `clearPoll` is called as a cleanup. The current placement in `handleBuild` is fine, but the description of the problem highlights a subtle interaction. The fix is to ensure `pollFailures` is reset *only* at the start of a new build attempt, which it currently is. The problem description is more about the *implication* of rapid re-clicks. The current code is acceptable.
