### BUG: Incorrect default value for ref state
**Severity**: low
**Location**: `RepoScanner`
**Problem**: The `ref` state is initialized to `'main'`, but the `onChange` handler for the ref input field sets it to `'main'` if the event target value is empty. This means if a user types something into the ref field and then deletes it, the `ref` state will correctly become `'main'`. However, if the user never interacts with the ref field and the initial `ref` state is `'main'`, the `trimmedRef` in `handleScan` will also correctly be `'main'`. The issue arises in the `onChange` handler for the ref input: `onChange={(e) => setRef(e.target.value || 'main')}`. If the user types "develop" and then deletes it, `e.target.value` becomes an empty string, and `setRef` is called with `'main'`. This is fine. However, the placeholder text for the ref input is "Branch (main)". If the user wants to explicitly scan the 'main' branch, they might type 'main'. If they then delete 'main', the `ref` state will correctly revert to 'main'. The problem is that the `ref` state is already initialized to `'main'`, so the `|| 'main'` in the `onChange` handler is redundant and could potentially mask issues if the initial state was intended to be an empty string representing "default". While not a critical bug, it's a slight inconsistency.
**Fix**: The `onChange` handler for the ref input should simply update the state with the input value, and the normalization to 'main' should primarily happen in `handleScan`.
