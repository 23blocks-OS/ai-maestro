### BUG: Unhandled promise rejection in `readJsonBody` and `readRawBody`
**Severity**: medium
**Location**: `readJsonBody` and `readRawBody` functions
**Problem**: The `Promise` constructors in `readJsonBody` and `readRawBody` attach `req.on('error', reject)` to the request stream. However, if an error occurs on the `req` stream *after* `req.on('end')` has already fired and the promise has been resolved (e.g., due to a network issue after the body has been fully received), the `reject` callback will be called on an already settled promise, leading to an unhandled promise rejection warning in Node.js. While not critical for correctness in most cases, it indicates a potential for noisy logs and unhandled exceptions.
**Fix**: Ensure that the `reject` callback is only called if the promise has not yet been resolved. This can be done by using a flag or by detaching the error listener once the promise is resolved.

```typescript
async function readJsonBody(req: IncomingMessage): Promise<any> {
  return new Promise((resolve, reject) => {
    const chunks: Buffer[] = []
    let resolved = false; // Flag to track resolution
    const onError = (err: Error) => {
      if (!resolved) {
        reject(err);
        resolved = true;
      }
    };
    req.on('data', (chunk: Buffer) => chunks.push(chunk))
    req.on('end', () => {
      if (resolved) return; // Already rejected by an error
      const body = Buffer.concat(chunks).toString('utf-8')
      if (!body) {
        resolve({});
        resolved = true;
        return;
      }
      try {
        resolve(JSON.parse(body));
        resolved = true;
      } catch (e) {
        reject(new Error('Invalid JSON body'));
        resolved = true;
      }
    })
    req.on('error', onError);
    // Optionally, remove error listener after end to prevent double-handling
    req.on('close', () => {
      req.off('error', onError);
    });
  })
}
```
