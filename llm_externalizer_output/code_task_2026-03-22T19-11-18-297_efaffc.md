### BUG: activeOps counter race condition
**Severity**: high
**Location**: buildPlugin, scanRepo, pushToGitHub
**Problem**: The `activeOps` counter is incremented at the beginning of the `try` block and decremented in the `finally` block. However, the `runBuild` function, which is launched asynchronously within `buildPlugin`, also performs operations that should ideally be counted towards `activeOps` if they are resource-intensive. As it stands, `activeOps` only counts the *launching* of the build process, not its actual execution. This means that multiple `runBuild` operations could be running concurrently without being limited by `MAX_CONCURRENT_OPS`, leading to resource exhaustion.

Furthermore, the `activeOps` counter is decremented in the `finally` block of `buildPlugin` *before* the asynchronous `runBuild` operation has completed. This means that the concurrency limit is released prematurely, potentially allowing more operations to start than the system can handle.

The `activeOps = Math.max(0, activeOps - 1)` line is also problematic. While `Math.max(0, ...)` prevents `activeOps` from going negative, it doesn't address the core race condition. If an operation fails *before* `activeOps` is decremented, or if multiple operations decrement it simultaneously, the counter could become inaccurate. A more robust approach would be to use a semaphore or a queue for managing concurrent operations.
**Fix**:
1. Introduce a separate mechanism for managing the concurrency of the `runBuild` function, or ensure `runBuild` is awaited if its resource usage should be limited by `MAX_CONCURRENT_OPS`. If `runBuild` is truly meant to be fire-and-forget from the perspective of `buildPlugin`'s `activeOps` count, then the current logic is fine for *launching* builds, but the overall system might still be vulnerable to resource exhaustion if `runBuild` itself is resource-intensive and not otherwise limited.
2. For `buildPlugin`, if `runBuild` is intended to be part of the `activeOps` limit, then `buildPlugin` should `await runBuild(...)` and the `activeOps` decrement should happen *after* the `await`. If `runBuild` is meant to run in the background, then `activeOps` should only count the "launching" phase, and `runBuild` itself needs its own concurrency management. Given the current `MAX_CONCURRENT_OPS` comment ("Max concurrent build/scan operations"), it implies that the *entire* build process should be limited, not just its initiation.

A better fix for `buildPlugin` would be to `await runBuild` and then decrement `activeOps` in the `finally` block. If `runBuild` *must* be asynchronous, then `activeOps` needs to be managed by `runBuild` itself, or a separate queueing mechanism should be used.
