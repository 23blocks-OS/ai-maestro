## 1. CURRENT ARCHITECTURE

### How Haephestos Currently Works

Haephestos is a **temporary, hidden agent** backed by a real Claude Code session running in a tmux session. It is not a real-time streaming connection but rather a **polling-based architecture** where:

1. **Session Creation**: A hidden tmux session (`_aim-creation-helper`) is created in the AI Maestro project directory
2. **Claude Launch**: Claude Code is launched with `--agent haephestos-creation-helper` using the Sonnet model
3. **Message Relay**: User messages from the chat UI are sent to the tmux session via `runtime.sendKeys()`
4. **Response Capture**: The terminal output is captured using `runtime.capturePane()` and parsed to extract Claude's responses
5. **Polling Loop**: The frontend polls the API endpoints to detect when Claude has finished responding

### Terminal Parsing & Pattern Detection

The service parses the Claude Code terminal output (from `capture-pane`) using several detection strategies:

**1. Response State Detection** (`detectResponseState` function):

- **Active tool call detection**: Scans for patterns like `+N more tool uses`, `ctrl+o to expand`, `ctrl+b ctrl+b`, and tool invocation lines (`ÔÄ┐ Read(`)
- **Separator line detection**: Finds lines with 10+ repeated characters (`ÔöÇ`, `ÔĽÄ`, `ÔĽÉ`) to delimit responses
- **Input prompt detection**: Checks for `>` or `ÔŁ»` prompt between separators to confirm response completion

**2. Config Block Extraction** (`parseConfigBlocks` function):
- **Strategy 1**: Regex detection of fenced code blocks with `json:config` language tag
- **Strategy 2**: Raw JSON array detection in terminal output (where markdown fences are stripped):
  - Finds lines starting with `[` (possibly after `ÔĆ║` response marker)
  - Tracks bracket depth to find complete JSON arrays
