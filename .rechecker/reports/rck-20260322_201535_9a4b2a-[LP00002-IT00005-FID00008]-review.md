### BUG: Inconsistent Error Handling for API Calls
**Severity**: medium
**Location**: `handleBuild`
**Problem**: The `handleBuild` function has inconsistent error handling for the initial build request and the subsequent polling requests. For the initial request, if `res.ok` is false, it attempts to parse `errData.error` from the response body. If this fails, it falls back to a generic `errorMessage`. However, the `catch` block for the initial `fetch` operation only sets a generic 'Failed to connect to server' message, without attempting to parse any error details from the response. This means if the server responds with a non-200 status but a valid JSON error message, the `catch` block will override it with a generic message, losing valuable debugging information.
**Fix**: The `catch` block for the initial `fetch` in `handleBuild` should be removed. Instead, the `if (!res.ok)` block should handle all non-successful responses, including network errors that might prevent `res.json()` from working. The current `if (!res.ok)` block already has a robust way to extract error messages.

### BUG: Potential Race Condition with `copyTimeoutRef`
**Severity**: medium
**Location**: `copyInstallCommand`
**Problem**: In the `copyInstallCommand` function, `copyTimeoutRef.current` is set to `null` inside the `setTimeout` callback. If a user clicks the copy button multiple times very quickly, it's possible for a new timeout to be set before a previous one has cleared `copyTimeoutRef.current` to `null`. While the `clearTimeout(copyTimeoutRef.current)` line attempts to prevent this, if the `setTimeout` callback executes *between* the `clearTimeout` call and the new `setTimeout` call, it could set `copyTimeoutRef.current` to `null` prematurely, leading to the new timeout not being cleared if the user clicks again. This is a minor race condition.
**Fix**: The `copyTimeoutRef.current = null` assignment should be moved outside the `setTimeout` callback and placed immediately after `setCopied(false)`. This ensures that the ref is cleared only when the "copied" state is reset, regardless of when the timeout callback actually executes relative to new clicks.
