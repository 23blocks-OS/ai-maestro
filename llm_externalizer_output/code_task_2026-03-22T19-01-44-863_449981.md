This means that errors originating from `runBuild` (e.g., if the `execPromise` inside `runBuild` rejects) are handled by `runBuild`'s internal `.catch` handler, but if an error occurs *before* `runBuild` is called, or if `runBuild` itself has an unhandled synchronous error, the outer `catch` block will handle it. Since `runBuild` is designed to run asynchronously and update the build status internally, the `activeOps` decrement should ideally happen in a `finally` block associated with the `runBuild` call itself, or the `buildPlugin` function should await `runBuild` if it intends to handle all its errors. Given the current design, the `activeOps` decrement in the `buildPlugin`'s outer `catch` block is correct for errors *before* `runBuild` is called. The `runBuild` function's `.finally` block should be removed, and `activeOps` should be decremented in the `buildPlugin`'s `finally` block, which would wrap the entire `try...catch` block.
