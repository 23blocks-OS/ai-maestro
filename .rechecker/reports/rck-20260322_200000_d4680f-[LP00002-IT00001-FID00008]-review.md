### BUG: Race condition in `pollAbortRef.current` assignment

**Location**: `handleBuild` function, specifically within the `setInterval` callback.
**Problem**: The logic intended to prevent race conditions with `pollAbortRef.current` is flawed.


When the `fetch` then completes, the `if (pollAbortRef.current === tickAbortCtrl)` check will be false because `pollAbortRef.current` is now `null`. This means `pollAbortRef.current` will not be reset to `null` by this specific tick's logic.
However, the problem is that if a *new* `handleBuild` call happens while a previous `fetch` is still in progress (e.g., due to a slow network or a rapid re-click), the new `handleBuild` will call `clearPoll()`, which will abort the *previous* `tickAbortCtrl` and set `pollAbortRef.current = null`. Then, the new `handleBuild` will immediately set `pollAbortRef.current = null` again before starting its own polling.
The core issue is that `pollAbortRef.current` is a shared mutable state that can be updated by `clearPoll` and by the polling interval's individual ticks. The current logic tries to ensure that `pollAbortRef.current` always points to the *currently active* `AbortController` for the *latest* fetch request.
The problem arises if `clearPoll` is called, nulling `pollAbortRef.current`, and then a *subsequent* tick of the *same* interval (which was already scheduled) runs. It will create a new `tickAbortCtrl`, assign it to `pollAbortRef.current`, and then fetch. If `clearPoll` is called again *before* this new fetch completes, it will abort this new fetch.
The comment `// If a new tick fires while this one is still awaiting, clearPoll() will // abort the current fetch via pollAbortRef.current before the ref is // replaced â€” preventing an un-abortable in-flight request.` describes the intent, but the implementation doesn't fully guarantee this. If `clearPoll` is called *again* before this new fetch completes, it will correctly abort it.
The actual bug is subtle: if `clearPoll()` is called, it sets `pollAbortRef.current = null`. Then, if a *stale* `setInterval` callback (from a previous `handleBuild` invocation that was not fully cleared before a new one started) executes, it will set `pollAbortRef.current` to its own `tickAbortCtrl`. This `tickAbortCtrl` will then *not* be aborted by the `clearPoll` call that initiated the new `handleBuild` because `pollAbortRef.current` was `null` at that time. This could lead to an un-aborted fetch request.

The current `clearPoll` already captures `pollAbortRef.current` in a local variable `abortCtrl` before nulling `pollAbortRef.current`. This is good.
The issue is more about the lifecycle of `pollAbortRef.current` within the `setInterval` callback.

The individual tick should only set it *before* the fetch and then rely on `clearPoll` to clean it up.

A more robust approach would be to ensure that `pollAbortRef.current` is *always* the `AbortController` for the *currently active* fetch.
