**Severity**: medium
**Location**: `RepoScanner` component, within the `scanResult.skills.map` function
**Problem**: The `key` prop for the mapped skill `div` is set to `skill.path`. While `skill.path` might be unique within a single scan result, it's not guaranteed to be globally unique across different repository scans or even if the same path appears in different branches/refs of the same URL. React keys should be stable and unique among siblings. ### BUG: `ref` state initialization and update logic
**Severity**: low
**Location**: `RepoScanner` component, `useState` for `ref` and `onChange` handler for the branch input.
**Problem**: The `ref` state is initialized to `'main'`. In the `onChange` handler for the branch input, `setRef(e.target.value || 'main')` is used. This means if a user types something and then deletes it, the `ref` will revert to `'main'`. While this might be the desired default behavior, the placeholder text for the input is "Branch (main)". If the user explicitly wants to scan a repository without specifying a branch (which might default to the repository's default branch on the server side, not necessarily 'main'), this logic forces 'main'. This could lead to an API call with 'main' when the user intended an empty or different default.
**Fix**: If the intention is to allow an empty branch field to signify the repository's default, the `onChange` handler should simply be `setRef(e.target.value)`. The placeholder "Branch (main)" already suggests 'main' as a common default. The server-side API should then handle an empty `ref` gracefully, perhaps by using the repository's actual default branch. If 'main' is strictly required as a fallback, the current logic is acceptable, but it's worth clarifying the intended behavior. For now, assuming the user should be able to clear the input, change `onChange={(e) => setRef(e.target.value || 'main')}` to `onChange={(e) => setRef(e.target.value)}`.
