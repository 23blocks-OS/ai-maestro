### 2.3 Task Assignment to Programmer Agents

**Assignment Workflow** (from `op-assign-task.md`):

```
1. QUERY READY TASKS
   - gh issue list --label status:ready --state open

2. SORT BY PRIORITY
   - Order: critical > high > normal > low

3. FILTER DEPENDENCIES
   - Skip tasks where blockedBy list is NOT empty

4. SELECT AGENT
   - Algorithm: skill match + load balancing

5. ASSIGN TASK
   a. Remove any existing assign:* label
   b. Add assign:<agent-name> label
   c. Update status:ready → status:in-progress
   
6. SEND NOTIFICATION
   - Send via agent-messaging skill (AMP)

7. WAIT FOR ACK
   - If no ACK within timeout, escalate/reassign

8. LOG ASSIGNMENT
   - Record in delegation log
```

**Agent Selection Algorithm**:
```
FOR EACH available agent:
  1. CHECK AVAILABILITY
     - Exclude: hibernated, offline agents
     
  2. CALCULATE SKILL MATCH SCORE
     - Match task type to agent specialization
     
  3. CHECK CAPACITY
     - 0-2 tasks: acceptable
     - 3+ tasks: at capacity (deprioritize)
     
  4. SORT BY:
     - Primary: skill match score (descending)
     - Secondary: current load (ascending)
     - Tiebreaker: recent similar task experience
```

| `draft` | `check_draft_gate(pr)` | PR is not a draft |
| `mergeable` | `check_mergeable_gate(pr)` | No merge conflicts |
| `tests` | `check_tests_gate(pr)` | All CI checks passing (statusCheckRollup conclusion == "SUCCESS") |
| `reviews` | `check_reviews_gate(pr, min_approvals)` | Required approvals present |
| `spec` | `check_spec_gate(pr, project_root)` | Feature PRs linked to approved spec |
| `issues` | `check_linked_issues_gate(pr)` | Linked to issues |

### 5.3 PR Workflow Managed by Integrator

**Lifecycle Steps**:

```
1. Poll → Run amia_orchestrator_pr_poll.py to get open PRs
2. Classify → Human PR = escalate; AI/bot PR = delegate directly
3. Delegate → Spawn subagent for review/changes
4. Monitor → Poll progress via background tasks (never block)
5. Verify → Run amia_verify_pr_completion.py before reporting
6. Report → Report to user, await merge decision
7. Handle failures → Delegate fixes
```

**Critical Rules**:
- Never block, write code, or merge without user approval
- Always verify before reporting status
- Escalate human PRs to user

### 5.4 Integrator Syncs with GitHub

**Script**: `amia_github_lifecycle_projects.py`

**Operations**:
- Move cards between columns via GraphQL
- Status field: "Status" (hardcoded)
- Column values: Dynamic from project configuration
- Sync direction: Local → GitHub

### 6.6 GraphQL Operations Required

```graphql
# Get Project
query GetProject($owner: String!, $repo: String!, $number: Int!) {
  repository(owner: $owner, name: $repo) {
    projectV2(number: $number) { id }
  }
}

# Get Fields
query GetFields($projectId: ID!) {
  node(id: $projectId) {
    ... on ProjectV2 {
      fields(first: 20) {
        nodes {
          ... on ProjectV2SingleSelectField {
            id, name, options { id, name }
          }
        }
      }
    }
  }
}

# Get Items
query GetItems($projectId: ID!) {
  node(id: $projectId) {
    ... on ProjectV2 {
      items(first: 100) {
        nodes {
          id
          content { ... on Issue { number title } }
          fieldValues(first: 8) {
            nodes {
              ... on ProjectV2ItemFieldSingleSelectValue { name optionId }
            }
          }
        }
      }
    }
  }
}

# Update Item
mutation UpdateItem($projectId: ID!, $itemId: ID!, $fieldId: ID!, $valueId: String!) {
  updateProjectV2ItemFieldValue(input: {
    projectId: $projectId
    itemId: $itemId
    fieldId: $fieldId
    value: { singleSelectOptionId: $valueId }
  }) {
    projectV2Item { id }
  }
}
```
