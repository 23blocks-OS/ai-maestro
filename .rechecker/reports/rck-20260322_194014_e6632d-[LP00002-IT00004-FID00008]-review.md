### BUG: Incorrect `clearPoll` call in `handleBuild`
**Severity**: medium
**Location**: `handleBuild`
**Problem**: In the `handleBuild` function, if the initial `fetch` call to `/api/plugin-builder/build` fails (i.e., `!res.ok`), the code sets an error, sets `building` to `false`, and then calls `clearPoll()`. However, `clearPoll()` is also called at the very beginning of `handleBuild`. If the initial fetch fails, no `setInterval` was ever started, so calling `clearPoll()` again is redundant and potentially misleading, as it resets `pollFailures.current` to 0 unnecessarily in a path where no polling occurred. While not critical, it's a slight logical inconsistency.
**Fix**: Remove the `clearPoll()` call from inside the `if (!res.ok)` block in `handleBuild`. The `clearPoll()` at the beginning of `handleBuild` is sufficient to clean up any previous intervals.

```typescript
// ...
      if (!res.ok) {
        const data = await res.json()
        setError(data.error || 'Build failed')
        setBuilding(false)
        // No interval was started in this path, so no need to clearPoll here.
        // The clearPoll() at the top of handleBuild already handled any previous intervals.
        // pollFailures.current is also reset at the top of handleBuild.
        return
      }
// ...
```
