### BUG: Incorrect `parseSessionName` logic for names containing numbers
**Severity**: medium
**Location**: `parseSessionName`
**Problem**: The regular expression `^(.+)_(\d+)$` in `parseSessionName` is too greedy. If an agent name itself contains numbers and underscores, it might incorrectly split the name. For example, `my_agent_v2_0` would be parsed as `{ agentName: "my_agent_v2", index: 0 }` instead of `{ agentName: "my_agent_v2_0", index: 0 }` if the intent was that `_0` is part of the agent's base name. The current logic assumes that the last `_` followed by digits always denotes the session index. This is problematic if an agent's base name can legitimately end with `_` followed by digits, and the session index is only implied when it's `_1` or higher. The examples provided (`"website" → { agentName: "website", index: 0 }`, `"website_0" → { agentName: "website", index: 0 }`) suggest that `_0` should be treated as part of the base name if it's the only suffix, or if the session index is 0, the suffix should be omitted. The current regex would incorrectly parse `website_0` as `{ agentName: "website", index: 0 }`, which matches the example, but it would also parse `my_agent_0` as `{ agentName: "my_agent", index: 0 }`. This is inconsistent with the `computeSessionName` function, which produces `"website"` for `("website", 0)` and `"website_1"` for `("website", 1)`. If `parseSessionName` receives `"website_0"`, it should ideally return `{ agentName: "website", index: 0 }` as per the example. However, if it receives `"my_agent_v2_0"`, and the agent's true name is `"my_agent_v2_0"` with session index 0, the current regex would incorrectly split it. The core issue is that `computeSessionName` omits `_0`, while `parseSessionName` tries to parse `_0` if present.
**Fix**: The `parseSessionName` function should be adjusted to align with the `computeSessionName` function's behavior, where an index of `0` does not result in a `_0` suffix. The regex should specifically look for a suffix `_` followed by a non-zero digit, or if the suffix is `_0`, it should be treated as if no suffix was present. A simpler and more robust approach is to only split if the last segment is `_` followed by a digit greater than 0.

```typescript
export function parseSessionName(tmuxName: string): { agentName: string; index: number } {
  if (!tmuxName) {
    return { agentName: '', index: 0 }
  }
  // Match a suffix like "_1", "_2", etc., but not "_0"
  const match = tmuxName.match(/^(.*)_([1-9]\d*)$/);
  if (match) {
    return { agentName: match[1], index: parseInt(match[2], 10) };
  }
  // If no such suffix, or if it ends with "_0", treat the whole name as agentName with index 0
  return { agentName: tmuxName, index: 0 };
}
```
