### BUG: Inconsistent Error Handling for Service Layer Failures
**Severity**: medium
**Location**: POST
**Problem**: The `POST` function has two different ways of handling errors originating from the `buildPlugin` service. If `buildPlugin` returns an object with an `error` property (e.g., `{ error: 'Validation failed', status: 400 }`), it's handled gracefully. However, if `buildPlugin` throws an unexpected exception, it's caught by the outer `catch` block, which returns a generic "Internal Server Error" with a 500 status. This inconsistency can obscure the true nature of errors from the service layer, making debugging harder. The `buildPlugin` function is expected to return a structured result, so a thrown error indicates an unexpected failure within the service itself, which should ideally be logged and potentially re-thrown or handled more specifically.
**Fix**: Ensure that the `buildPlugin` service consistently returns a structured result object, even for unexpected internal errors. The API route should then only need to check the `result.error` property. If `buildPlugin` is designed to throw exceptions for unrecoverable errors, the `catch` block should log the error for debugging purposes before returning a generic 500.

```typescript
/**
 * Plugin Builder - Build API
 *
 * POST /api/plugin-builder/build - Start a plugin build
 *
 * Thin wrapper: all validation lives in the service layer.
 */

import { NextResponse } from 'next/server'
import type { NextRequest } from 'next/server'
import { buildPlugin } from '@/services/plugin-builder-service'

export async function POST(request: NextRequest) {
  // Parse the request body separately so a JSON parse failure is always a
  // client error (400), while errors from the service layer return 500.
  let body: unknown
  try {
    body = await request.json()
  } catch {
    return NextResponse.json(
      { error: 'Invalid request body' },
      { status: 400 }
    )
  }

  try {
    const result = await buildPlugin(body)

    // The service layer is expected to return a structured result,
    // including errors. If result.error is present, it's a handled error.
    if (result.error) {
      // Log the error for server-side visibility, but return the client-facing error.
      console.error('Plugin build service error:', result.error);
      return NextResponse.json(
        { error: result.error },
        { status: result.status }
      )
    }
    return NextResponse.json(result.data, { status: result.status })
  } catch (error) {
    // This catch block should ideally only be hit for unexpected exceptions
    // thrown by the buildPlugin service, not for expected validation/business logic errors.
    // Log the unexpected error for debugging.
    console.error('Unexpected error during plugin build:', error);
    return NextResponse.json(
      { error: 'Internal Server Error' },
      { status: 500 }
    )
  }
}
```
