### BUG: Incorrect `activeOps` Decrement in `buildPlugin`
**Severity**: critical
**Location**: `buildPlugin`
**Problem**: The `activeOps` counter is intended to track currently active operations. In `buildPlugin`, `activeOps` is incremented, and then `runBuild` is called as fire-and-forget. The `finally` block attached to the `runBuild` promise is responsible for decrementing `activeOps` when the asynchronous build operation completes. However, the `buildPlugin` function also has its own `finally` block which conditionally decrements `activeOps` based on the `opStarted` flag.

The logic for `opStarted` is flawed. `opStarted` is set to `true` *after* `activeOps` is incremented. If an error occurs *after* `activeOps` is incremented but *before* `runBuild` is called (e.g., `fs.mkdir` fails), the `buildPlugin`'s `finally` block will decrement `activeOps` because `opStarted` will be `true`. This is correct.

However, if `runBuild` is successfully launched, `opStarted` will be `true`. The `buildPlugin`'s `finally` block will then *not* decrement `activeOps` (because `if (!opStarted)` is false). This means that if `runBuild` is successfully launched, the `activeOps` counter is *only* decremented by the `runBuild`'s `finally` block.

The problem arises if `runBuild` itself throws an error *before* its `finally` block is reached (e.g., if the `execPromise` call within `runBuild` rejects, and the `.catch` handler for `runBuild` is executed, but the `.finally` is somehow bypassed or delayed in a way that causes a race condition or an unhandled rejection before the decrement). More importantly, the current structure implies that `activeOps` is incremented in `buildPlugin` and *then* decremented in `runBuild`'s `finally`. The `buildPlugin`'s `finally` block should *only* decrement if `runBuild` was *not* successfully launched.

The current implementation of `buildPlugin`'s `finally` block is:
```typescript
finally {
  if (!opStarted) { // This condition is problematic
    activeOps = Math.max(0, activeOps - 1)
  }
}
```
This means if `opStarted` is `true` (which it will be if `activeOps++` was executed), the `buildPlugin`'s `finally` block will *not* decrement `activeOps`. This is the intended behavior, as `runBuild`'s `finally` is supposed to handle the decrement.

The actual bug is in the `runBuild`'s `.finally()` block. It's attached to the *return value* of `runBuild`, which is a `Promise<void>`. If `runBuild` itself throws a synchronous error (which is unlikely given it's an `async` function, but possible if the `execPromise` setup itself fails synchronously before returning a promise), or if the promise chain within `runBuild` is somehow broken, the `finally` might not execute reliably.

The primary issue is that `activeOps` is incremented in `buildPlugin` and then `runBuild` is called. The decrement should *always* happen when the *asynchronous operation completes*. The `buildPlugin` function returns a `ServiceResult` *before* the build completes. Therefore, the `activeOps` decrement *must* be tied to the completion of `runBuild`, not the return of `buildPlugin`.

The current code has a subtle bug:
1. `activeOps++` happens in `buildPlugin`.
2. `runBuild` is called.
3. `buildPlugin` returns.
4. `runBuild`'s `.finally()` is supposed to decrement `activeOps`.

The `buildPlugin`'s `finally` block is trying to handle early errors, but its condition `if (!opStarted)` is inverted for its purpose. `opStarted` is set to `true` *after* `activeOps` is incremented. So, if `activeOps` was incremented, `opStarted` is `true`, and the `buildPlugin`'s `finally` block will *not* decrement. This means that if `runBuild` is successfully launched, the `activeOps` counter is *only* decremented by the `runBuild`'s `finally` block. This is the correct design.

The bug is actually in the `runBuild`'s `.catch` block. If `runBuild` fails, its `.catch` block updates the build status, but the `.finally` block is *still* responsible for decrementing `activeOps`. The current structure is:
```typescript
runBuild(...)
  .finally(() => {
    activeOps = Math.max(0, activeOps - 1)
  })
  .catch(err => {
    // ... update status ...
  })
```
This is correct. The `.finally` block will always execute after the promise settles, regardless of success or failure.

The only potential bug here is if `runBuild` itself throws a synchronous error *before* returning a promise, which would prevent the `.finally` from ever being attached. However, `runBuild` is an `async` function, so it *always* returns a promise. Any synchronous errors inside `runBuild` would cause the returned promise to reject.

`opStarted = true`

`fs.mkdir` throws.

`buildPlugin`'s `finally` block: `if (!opStarted)` is `false`, so it does *not* decrement. This is a bug. `activeOps` was incremented, but no asynchronous operation was launched to decrement it.

**Fix**: The `opStarted` flag should be used to determine if `activeOps` was incremented *and* if `runBuild` was successfully launched. If `activeOps` was incremented but `runBuild` was *not* launched, then `buildPlugin`'s `finally` should decrement.

  opStarted = true;

  // AND runBuild was NOT successfully launched,
  // then this finally block needs to decrement activeOps.
  if (opStarted && !runBuildLaunched) {

  }
