### BUG: Incorrect `portable_sed` usage for `safe_dir`
**Severity**: high
**Location**: `act4_start_and_register`
**Problem**: The `portable_sed` function is called with `s#{{INSTALL_DIR}}#${safe_dir}#g` where `safe_dir` is already escaped for `sed` metacharacters. However, `portable_sed` itself uses `sed -i '' "$@"` or `sed -i "$@"`. The `s#...#...#g` syntax uses `#` as a delimiter. If `safe_dir` contains a `#` character, it will prematurely terminate the `sed` command's substitution pattern, leading to a syntax error or incorrect substitution. The current escaping `sed 's/[[\.*^$#&\\\/]/\\&/g'` escapes `#`, which is correct for the `s#...#...#g` delimiter, but the `|` delimiter used for `VERSION` and `SELECTED_GATEWAYS` is not escaped. This creates an inconsistency and potential for issues if `INSTALL_DIR` contains `|`. More critically, the `safe_dir` variable is already escaped for *all* `sed` metacharacters, including the `#` delimiter. When `sed` then tries to interpret `s#{{INSTALL_DIR}}#${safe_dir}#g`, the escaped `#` in `safe_dir` will be treated as a literal `#` and not a delimiter, which is correct. The problem is that the `portable_sed` function itself doesn't handle the delimiter escaping, it just passes the arguments. The current escaping for `safe_dir` is actually *too aggressive* for the `#` delimiter if the intent was for `safe_dir` to be used as a literal string within the replacement part. However, the primary issue is that `portable_sed` is called with different delimiters (`#` and `|`) without ensuring the `safe_dir` is correctly escaped for *all* potential delimiters used in the `sed` calls. The `safe_dir` is escaped for `|` as well, so `portable_sed "s|{{VERSION}}|$VERSION|g"` would also work with `safe_dir` if it were used there. The main problem is the `safe_dir` escaping is done once, but then used with two different delimiters (`#` and `|`) in the `portable_sed` calls. The current escaping `sed 's/[[\.*^$#&\\\/]/\\&/g'` correctly escapes `|` as well, so it should be fine. The bug is more subtle: if `INSTALL_DIR` itself contains a `\` character, the `safe_dir` will have `\\` for each `\`. When `sed` processes `s#...#${safe_dir}#g`, `\\` in `safe_dir` will be interpreted as a literal `\` followed by a literal `\`. This is usually the desired behavior for paths. The actual bug is that the `portable_sed` function is called with a replacement string (`${safe_dir}`) that has already been escaped for `sed` metacharacters. `sed` will then interpret these backslashes again. For example, if `INSTALL_DIR` is `/tmp/my_dir`, `safe_dir` will be `/tmp/my_dir`. If `INSTALL_DIR` is `/tmp/my\dir`, `safe_dir` will be `/tmp/my\\dir`. When `sed` sees `s#...#${safe_dir}#g`, it will interpret `\\` as a single `\`. This means if the original path had a backslash, it would be unescaped by `sed`. This is a common pitfall when pre-escaping strings for `sed` replacement.
**Fix**: The `safe_dir` variable should be escaped in a way that `sed` interprets it literally. A common way to do this is to escape `&` (which `sed` uses for back-references) and the delimiter itself. Since the delimiter is `#`, we need to escape `#` and `&`. The current escaping is too broad. A simpler and more robust approach is to use a different `sed` command or to carefully construct the replacement string. A common pattern is to use `printf '%s\n' "$INSTALL_DIR" | sed -e 's/[\/&]/\\&/g'` to escape the path for use as a replacement string, then use `sed "s#{{INSTALL_DIR}}#$(escaped_path)#g"`. However, the current `portable_sed` function is designed to take the entire `sed` command as an argument. The most direct fix is to ensure that the `safe_dir` variable is escaped correctly for the *replacement part* of a `sed` substitution, which means escaping `&` and the chosen delimiter (`#`).
