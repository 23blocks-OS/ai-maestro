### BUG: Inconsistent `enabledCount` update logic
**Severity**: medium
**Location**: `togglePlugin`
**Problem**: The `enabledCount` for a marketplace is updated optimistically based on whether the plugin is found within that specific marketplace's plugin list. However, the condition `m.plugins.some(p => p.key === key)` is redundant and potentially incorrect. If the plugin `p.key === key` is being toggled, it *must* belong to one of the marketplaces. The current logic implies that a plugin might be toggled but not belong to any marketplace in the `prev` state, which contradicts the fact that `m.plugins.map` is iterating over plugins within `m`. More critically, if a plugin is toggled, its `enabledCount` should be updated in the *specific* marketplace it belongs to, not conditionally based on `m.plugins.some`. The current logic would add or subtract 1 from `m.enabledCount` for *every* marketplace that contains the plugin, which is incorrect if a plugin can exist in multiple marketplaces (though the current data structure suggests a plugin belongs to a single marketplace). Assuming a plugin belongs to only one marketplace, the `enabledCount` update should only happen for that specific marketplace.
**Fix**: The `enabledCount` update should be part of the `map` operation for the specific marketplace that contains the plugin being toggled. The `m.plugins.some` check within the `enabledCount` calculation is unnecessary and could lead to incorrect counts if a plugin is somehow duplicated across marketplaces (which should ideally not happen).

```typescript
// Inside togglePlugin
```

The fix ensures that `enabledCount` is only adjusted for the marketplace that actually contains the plugin being toggled.

### BUG: Potential for stale `enabledCount` in `togglePlugin` optimistic update
**Severity**: low
**Location**: `togglePlugin`
**Problem**: In the `togglePlugin` function, the optimistic update for `enabledCount` relies on `m.enabledCount + (m.plugins.some(p => p.key === key) ? (currentEnabled ? -1 : 1) : 0)`. While the previous bug addresses the `m.plugins.some` condition, there's a more subtle issue. If `m.enabledCount` itself was already stale (e.g., due to a previous optimistic update that failed to be reconciled with the server, or if another client modified the state), this calculation will compound the error. The `enabledCount` should ideally be derived from the `plugins` array itself after the optimistic update, or the server should return the updated `enabledCount` for reconciliation.
**Fix**: Instead of directly incrementing/decrementing `m.enabledCount`, recalculate it based on the updated `plugins` array for that marketplace. This ensures consistency between the `plugins` array and `enabledCount`.

```typescript

  const updatedPlugins = m.plugins.map(p => p.key === key ? { ...p, enabled: !currentEnabled } : p);
  const newEnabledCount = updatedPlugins.filter(p => p.enabled).length;

  return {
    ...m,
    plugins: updatedPlugins,
    enabledCount: newEnabledCount, // Recalculate based on updated plugins
  };
}));
// The totals.enabledPlugins update is fine as it's a global count.
setTotals(prev => ({ ...prev, enabledPlugins: prev.enabledPlugins + (currentEnabled ? -1 : 1) }));
```
This ensures that `enabledCount` is always consistent with the `plugins` array within each marketplace object after an optimistic update.
